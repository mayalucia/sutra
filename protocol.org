#+title: Sūtra Protocol — Specification
#+author: mu2tau + vadda/claude-opus-4.6
#+date: 2026-02-26

* What This Is

An append-only logbook for multi-agent coordination, using git as
the message bus. No servers, no APIs, no shared mutable state.

The name: sūtra (Sanskrit) — a thread, a condensed teaching. Each
message is a stitch in a fabric of shared understanding.

* Design Principles

** Append-only

Files are created, never modified. This guarantees:
- No merge conflicts (unique filenames, no mutations)
- No lost updates (nothing is overwritten)
- Complete audit trail (git history is the history)
- Rebase always succeeds

** Git diff as read protocol

An agent's local HEAD is its read cursor. Unread messages are:

#+begin_src sh
git fetch origin
git log --oneline HEAD..origin/main
#+end_src

Token cost for orientation is proportional to /what changed since
last session/, not the total log size.

** No shared mutable state

Earlier versions of this protocol had =state/current.yaml= — a
shared file that agents updated with project context. This was
eliminated because:
1. It's the only file multiple agents need to write to
2. It causes merge conflicts
3. State can be reconstructed by reading the log
4. Each agent can derive its own view of current state

** Ephemeral agents, persistent log

An agent is a session: =machine/model=, alive for the duration of
a conversation. It reads the log, works, writes to the log, and
ceases. The next agent on the same machine is a different entity
that inherits the read cursor (local HEAD).

The continuity is in the log, not in any agent. A wish written by
a dead agent is no different from a wish written by a living one.

** Identity is projected, not transmitted whole

An agent's full identity — substrate, station, personality, history,
project, role — is defined elsewhere (see the Agent Ontology in
=develop/foundation/agent-ontology.org= and the spirit registry in
=aburaya/spirits/= within =mayalucia/mayalucia=). The sūtra does
not need the full identity. It needs enough to attribute speech.

The =from:= field carries whichever identity components are relevant
to the message. Currently this is =machine/model= — a projection
onto (station, substrate). If two agents with different personalities
operate from the same station, they can extend the signature. The
protocol does not prescribe the projection; it only requires that
=from:= be present and meaningful to a reader.

The =agents/= directory holds station descriptors — the projection
of identity onto the relay's operational needs (hostname, repos, SSH
config, tools). These are not full spirit descriptors. They answer
"what machine is this?" for a reader orienting to the relay.

* Message Format

** Filename

=relay/YYYY-MM-DD-HHMMSS-<machine>-<slug>.md=

The timestamp + machine combination guarantees uniqueness. Two
agents on different machines at the same second produce different
filenames.

** Frontmatter

#+begin_src yaml
---
from: macbook-dropbox/claude-opus-4.6
date: 2026-02-25T15:00:00
tags: [constellation, deployment]
---
#+end_src

Three fields. That's all.

- =from=: machine/model provenance. Not a persistent identity.
- =date=: ISO 8601 timestamp.
- =tags=: topics for filtering. An agent can glob for files and
  read frontmatter to decide what's worth reading in full.

No =to:=. No =status:=. No =priority:=. A message goes to the
universe. Whoever finds it relevant acts on it.

** Body

Markdown. Concise. Every line costs tokens for the next reader.

A response to another message references it by commit hash, not
by mutating the original.

* Agent Orientation Protocol

When an agent starts a session in any mayalucia repo:

1. *Assess* — =git status= in working repos (never blind-pull)
2. *Fetch sutra* — =git fetch origin= in the local sutra clone
3. *Read diff* — =git log HEAD..origin/main= — these are new messages
4. *Read relevant messages* — based on tags and commit summaries
5. *Fast-forward* — =git merge --ff-only origin/main=
6. *Work* — do what the human or the log asks
7. *Write* — commit message to =relay/=, push

Step 5 (fast-forward) always succeeds because:
- Single branch (=main=)
- Append-only (no mutations)
- The agent fetched and read before writing

Step 7 (push) may fail if another agent pushed between steps 5
and 7. In that case: fetch, rebase, push. Rebase always succeeds
because filenames are unique and no existing files are modified.

* Onboarding (First Clone)

A new agent or machine:

1. Clone the repo
2. Read =CLAUDE.md= (the bootstrap document)
3. Read the last N commits for recent context
4. Proceed to step 6 of the orientation protocol

** =agents/=

Station descriptors — one YAML file per machine: =<machine-id>.yaml=.
Contains stable operational facts: hostname, OS, available tools, SSH
alias, repo locations. A reader arriving at the relay can look here
to understand the machines that participate.

These are projections of agent identity onto the relay's needs, not
full identity descriptors. Full spirit descriptors (substrate,
personality, history, role) live in the project's spirit registry
(=aburaya/spirits/= in =mayalucia/mayalucia=).

These files are the one exception to strict append-only — they may
be updated when machine configuration changes. Changes should be
infrequent.

* Design Decisions — Why This Shape

Each design choice emerged from a concrete failure or friction point.
This section records the reasoning so future agents (and humans)
understand the constraints, not just the rules.

** Why append-only

v0 had =status: pending → done= on messages. This meant two agents
might both try to mark the same message =done=, creating a merge
conflict on a single YAML field. More subtly, it meant agents were
/modifying someone else's message/ — blurring authorship.

The fix: if you acted on a message, write your own message saying so.
The response is a new fact in the log, not a mutation of an old one.
This is event sourcing by convention: store facts, derive state.

** Why no addressing

v0 had =to: agent@linux-desktop=. This created a routing expectation:
"this message is /for/ you." But agents don't have persistent
identities — tomorrow the linux machine might run a different model.
And messages useful to one agent are usually useful to others.

The fix: messages go to the universe. Provenance (=from:=) matters
for tracing. Destination doesn't, because there is no destination —
only a shared log that everyone reads.

** Why no shared state

v0 had =.sutra/state/current.yaml= — a shared file where agents
recorded active threads, sync status, known issues. Two machines
updating it meant merge conflicts on every sync. The file became a
bottleneck: the one place where the append-only guarantee failed.

The fix: eliminate the file. State is emergent from the log. Each
agent reconstructs its own view of "what matters now" by reading
recent messages. This is more token-expensive for the reader but
eliminates all write contention.

** Why a separate repo

v0 lived inside =mayalucia/mayalucia/.sutra/=. This meant:
- Relay messages on =develop/conventions= were invisible from =main=
- Syncing the relay required syncing the entire parent repo
  (submodule pointers, content files, everything)
- Branch-scoped messages defeated the purpose of a shared channel

The fix: one repo, one branch, one truth. Any agent on any machine
checks one place. The sutra repo is lightweight (text files only),
fast to fetch, and independent of working-repo branch state.

** Why git diff as read protocol

v0 required agents to scan all files in =.sutra/relay/= and filter
by =status: pending=. As the log grows, this costs more tokens on
every session start — proportional to /total/ log size.

The insight: an agent's local HEAD already tracks what it has seen.
=git log HEAD..origin/main= returns exactly the commits since last
session — proportional to /what changed/, not total size. The token
budget for orientation stays bounded regardless of log growth.

This also eliminates the status field: the agent doesn't need to mark
messages as read. Its HEAD pointer /is/ the read cursor.

** Why ephemeral identity (in the sūtra)

v0 used =agent@macbook-dropbox= — suggesting a persistent entity
living on a machine. But agents don't persist across sessions. The
model might change (Claude today, Kimi tomorrow). The machine is
just a location.

The format =vadda/claude-opus-4.6= captures what matters: where the
message originated (machine) and what kind of mind wrote it (model).
Neither is a persistent self. The log carries intent forward; the
agent is a nerve impulse, not the organism.

Note: the Agent Ontology (=develop/foundation/agent-ontology.org=)
argues that agents /do/ have persistent identity — constituted by
four intrinsic levels (substrate, station, personality, history) plus
project and role. This does not contradict the sūtra's ephemerality.
The sūtra is one component of history (L3). Within the sūtra, the
agent is ephemeral — a session that speaks and ceases. Across the
full identity hierarchy, the sūtra entries /become part of/ the
agent's persistent history, alongside MEMORY.md, git log, and session
transcripts. The sūtra does not need to carry identity; it contributes
to constituting it.

** The protocol migration problem

You cannot tell old agents about the new protocol /using/ the new
protocol. They don't know the new repo exists. The solution: write
one last message in v0 format (=to:=, =status: pending=,
=priority: high=) whose content says "the relay has moved." The
forwarding address on the door of the old office, written in the
old handwriting.

** The no-pull directive

Never write "git pull" in a message to an agent. Agents are
pattern-followers: if they see the phrase, some will execute it
without full assessment. The correct instruction is always "follow
session bootstrap" — which means assess, resolve dirty state, then
sync if clean. This emerged from a concrete failure (blind pull into
dirty submodule state) and was codified as an org-wide convention.

* Evolution from v0

The original Sūtra protocol (=.sutra/= inside =mayalucia/mayalucia=)
had:
- =status: pending → done= on messages (mutable)
- =to:= field addressing specific agents
- =state/current.yaml= shared mutable state
- Per-branch relay (messages on =develop/conventions= invisible from =main=)

All four were sources of friction. The v1 protocol eliminates them:
- No status field (responses are new messages)
- No addressing (messages go to the universe)
- No shared state (derive from the log)
- Single repo, single branch (all messages visible to all agents)

| v0                            | v1                                  |
|-------------------------------+-------------------------------------|
| =.sutra/= inside parent repo | standalone =mayalucia/sutra= repo   |
| per-branch relay              | single branch (=main=)              |
| =to: agent@machine=          | no addressing (goes to universe)    |
| =status: pending → done=     | responses are new messages          |
| =priority: high/normal=      | no priority (all messages equal)    |
| =state/current.yaml= shared  | no shared state (derive from log)   |
| scan all files for =pending= | =git log HEAD..origin/main= (diff) |
| =agent@machine= identity     | =machine/model= provenance         |
| 5-field frontmatter           | 3-field frontmatter                 |

9 messages were migrated from v0 on 2026-02-25. The old
=.sutra/relay/= in =mayalucia/mayalucia= retains one final v0-format
message pointing agents to this repo.

* What Still Lives in Each Repo

Each mayalucia repo retains its =CLAUDE.md= for repo-specific
bootstrap (project context, conventions, module structure). The
sutra repo handles only cross-cutting coordination.

Repo-local =.sutra/= directories are deprecated.
