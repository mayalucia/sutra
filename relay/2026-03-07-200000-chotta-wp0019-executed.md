## WP-0019 executed: LLM adapter and QA scenario

String model resolution, LLM QA benchmark, and evaluate() extended
to handle non-numeric domains.

### What was built

- **Model resolver** (`src/dmt/models/`): `resolve("echo")`,
  `resolve("anthropic/claude-haiku-4-5-20251001")`, `resolve("openai/gpt-4o")`
  — string specs to callable model objects with `.name` and `.predict()`.
- **Three offline baselines**: Echo (returns question), Random (picks
  from answer pool), Template (rule-based pattern matching).
- **Anthropic + OpenAI adapters**: lazy client init, latency tracking,
  token counting.
- **LLM QA scenario**: 38 curated QA pairs across 4 categories
  (factual, reasoning, coding, creative) at 3 difficulty levels.
- **LLM metrics** (`src/dmt/metrics/llm.py`): exact_match, fuzzy_match,
  mean_latency.
- **evaluate() extended**: accepts string model specs; auto-detects
  numeric vs string data and switches between RMSE/bias/skill and
  exact_match/fuzzy_match accordingly.
- **CLI updated**: `dmt eval --scenario llm_qa` works with offline
  baselines. Also added `equity_forecast` to CLI registry.
- **36 tests**: resolver, dataset, baselines, metrics, end-to-end
  pipeline, string spec integration. API tests marked `@pytest.mark.llm`.

### Architectural note

The evaluate engine now has two metric modes: numeric (RMSE/bias/skill)
and string (exact_match/fuzzy_match), chosen by dtype detection. This
was necessary to run QA through the same `evaluate()` pipeline without
refactoring the metric system (deferred to future work). The approach
is backward-compatible — all existing numeric scenarios work unchanged.
